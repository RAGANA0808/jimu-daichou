# セキュリティ・プライバシー方針

寺務台帳は **個人情報・宗教情報・死者の情報** という、特に機微な情報を扱う。本ドキュメントでは、設計・運用上遵守すべきルールを定める。

---

## 扱う情報の機微性

| 情報 | 機微性 | 理由 |
| :--- | :--- | :--- |
| 氏名・住所・電話・メール | 個人情報 | 個人情報保護法の保護対象 |
| 家族構成・続柄 | 個人情報 | 第三者に知られたくない家族事情を含むことがある |
| 宗教的帰属（檀家関係） | **要配慮個人情報** | 個人情報保護法上、本人同意なしの取得禁止 |
| 過去帳（故人情報・戒名） | 要配慮個人情報＋歴史的価値 | 故人の尊厳・家族の心情・文化財的価値 |
| 会話履歴（伝言メモ） | 機微情報 | 相談内容・悩み事を含むことがある |
| 会計情報（お布施額等） | 経済情報 | 個人の金銭状況に関する情報 |

---

## 設計原則

### 1. 最小権限の原則
- 未ログインで取得できる情報はゼロ（ログイン画面・規約ページ以外）。
- ユーザーは自分が所属するテナントの情報のみアクセス可能。

### 2. 多層防御
- **層 1: 認証** (Supabase Auth)
- **層 2: DB の RLS** (他テナント行が物理的に読めない)
- **層 3: アプリ層の `withTenant()` ラッパー**
- **層 4: UI で権限別の表示制御**

### 3. ログへの個人情報出力禁止
- 氏名・住所・電話・メールはログに出さない。
- デバッグ時も ID (`personId: abc-123`) のみ出す。
- 構造化ログで `PII` フィールドを自動マスクするラッパーを `lib/logger/` に整備予定。

### 4. 過去帳は物理削除禁止
- `deletedAt` カラムによる論理削除のみ。
- Prisma レベルでも `death_ledger_entries` への DELETE を発行しない規約。
- 管理画面に「削除」ボタンを作らない。代わりに「非表示化」「訂正履歴」を実装。

### 5. 通信の暗号化
- HTTPS 必須。ローカル開発以外は HTTP 不可。
- 本番は HSTS ヘッダ配信。

### 6. 保存時の暗号化
- PostgreSQL ボリュームは at-rest 暗号化（Supabase 標準）。
- Storage (Supabase) も at-rest 暗号化。
- **特に機微な列（例: 個別の相談内容メモ）は将来的にアプリ層暗号化を検討**。

---

## バックアップ

- Phase1: Supabase 自動バックアップ (Point-in-Time Recovery)。
- 定期的にエクスポートスクリプトでローカルに暗号化バックアップを取る（運用整備後）。
- バックアップファイルは `.gitignore` で除外、機微 CSV のパターンは監査。

---

## 第三者提供

- 本プロダクトは第三者への個人情報提供を行わない。
- Google Calendar 連携は「ユーザー自身の Google アカウントへの登録」であり、第三者提供ではない。

---

## 事故対応（準備）

- データ漏洩が疑われた場合の手順を Phase2 以降に `docs/runbooks/incident.md` として整備する。
- Phase1 時点では、事故発生時に即座にデータベースを隔離し、開発者のみで対応する想定。

---

## 禁止事項 (CLAUDE.md §7 より引用)

- ❌ `tenantId` 条件を含まない DB クエリ
- ❌ RLS ポリシーの無効化
- ❌ 個人情報を含むログ出力
- ❌ 過去帳の物理削除
- ❌ 過去帳・檀信徒データの公開リポジトリへの混入
