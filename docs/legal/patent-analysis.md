## 4. 回避設計の指針（同じ課題を別手段で解く）

原則: **請求項の「特定の手段・組合せ」を再現しない**。課題（顧客一元管理・カルテ・年忌自動化）の達成は問題ないが、クレームされた具体的構成と一致させない。請求項全文確定までは保守的に設計する。

### E11 ダッシュボード / E02 タイムライン（対 せいざん P1）
- P1 請求項の核心は「**第1情報の更新を契機に、第2情報（契約情報）を更新された順番でポータルに動的配列表示**」。
- 回避:
  - ダッシュボードの主軸ソートを「**更新順**」一本にしない。**期限ベース（命日前/法要日前/契約期限/未納）の "気づき" カードを主構成**にする（バックログ E11 の方針と一致）。
  - 対応履歴タイムラインは「人物カルテ配下の時系列ノート」であり、契約情報を更新順にポータル配列する UI とは別物として明確に分離する。
  - 「固有情報の更新 → 契約情報の自動並べ替え」という **更新契機の自動再配列**を必須挙動にしない（任意ソートの一選択肢に留める）。

### E24 見込み客ステータス（対 せいざん P1/P2）
- P1/P2 は「**見込み客＋契約情報をリスト状ポータルで更新順表示**」の組合せが核心。
- 回避: 見込み客は **属性タグ/ステータス**として既存 Household/Person モデルに持たせ（バックログ通り）、「見込み客固有情報＋契約情報を更新順ポータル表示」という P1 と同型の専用画面を作らない。見込み客一覧は通常の検索/フィルタ結果として提供する。

### E24/E33 施主の代替わり（対 TERA P5 — 最重要）
- P5 のプレス明記クレーム候補「**施主が亡くなった際に次の施主を自動的に設定**」。
- 回避（請求項確定までの保守設計）:
  - **「死亡を契機に次施主を自動確定する」全自動ロジックは実装しない**。
  - 代わりに、**人手承認を介在させる承継ウィザード**にする: 死亡登録時は「承継候補の提示・通知」までに留め、次施主の確定は **担当者の明示的操作（承認）**を必須にする。自動「設定」ではなく手動「承認」という構成差を設ける。
  - 承継の連鎖は時系列で記録（バックログ E10/E24 と整合）しつつ、トリガは死亡イベントの自動連鎖ではなくユーザー操作起点とする。

### E03 案内発送 / E15 通知（対 TERA P5・Classix P6）
- 回避: 発送・通知は **手動トリガ（対象抽出→確認→送信）を基本**とし、「死亡/命日イベント → 自動発送」の全自動パイプラインを既定にしない。年忌対象抽出は計算結果の提示に留め、送信は人手確認ステップ（バックログ E03 の「送信前プレビュー・確認」と一致）を必ず挟む。

### E05 中陰 / E22 弔い上げ（対 Classix P6・R1 公知）
- 回避: 中陰・年忌の日付算出は **純粋計算ロジック（`lib/chuin/`・`lib/nenki/`）として独立**させ、R1 の公知計算と同等であることを明確化（防御）。**オンライン参拝・お布施決済・仮想寺院・納骨箱搬送と連結しない**（P6 の 6 工程構成と一致させない）。通知は寺側内部リマインドに限定。

### E09 墓地台帳・地図（対 P7〜P10）
