## 1. 情報アーキテクチャ（グローバルナビ）

ヘッダー: ロゴ「寺務台帳 檀家・墓・納骨堂クラウド管理」／右上に `▶メモ`・`担当者名 | 寺院名`・`ログアウト`。

水平トップナビ（タブ）:
```
案件 | 檀信徒 | 過去帳 | 法要 | 区画 | お壇 | 契約 | 合祀 | 行事 | 棚経 | お手紙 | 会費 | 帳票 | 集計・分析 | 設定
```

**示唆（我々への反映）:**
- 現状の我々のナビ（ダッシュボード／檀信徒／過去帳／法要／区画／会計／設定）より粒度が細かく、業務の名詞そのものがメニューになっている。特に **「案件」（=問い合わせ〜契約前の見込み客ファネル）が最上位の第一タブ** に置かれている点が重要。寺院運営は「新規問い合わせ→見学→契約」の営業フローを持つことを物語る。
- 「お手紙」（案内状差込印刷）「会費」「帳票」「集計・分析」「合祀」「棚経」が独立タブ。我々の `feature-backlog` の E08/E10/E14 等と対応。
- ただし15タブは多すぎてPCでも横に詰まっている。**我々は「次世代・スマホ両対応」方針なので、ここはそのまま真似ず、グルーピング（カルテ／法務／墓地／会計／運用）した方が良い**。

## 2. 案件一覧（フレーム10, 82:00）— 中核の作業ビュー

CRM のリードリストに近い。**おそらく日常で最も開く画面**。

- **検索バー**: `キーワード（案件名・顧客氏名・TEL・照合番号）` を最上段に常設。
- **フィルタ**: `履歴` / `全て・未契約・契約済`（ラジオ）/ `☆のみ` / `伝言` / `予定` / `全担当（ドロップダウン）`。
- **タグ絞り込みチップ**: `合祀予定` `仮予約` `確認事項あり` `高齢夫婦` `心配` `全て表示`。＝運用者が自由に付けた状況タグで横断抽出。
- **日付レンジ**: `問い合せ日` 〜 / `契約日` 〜。
- **案件カード**（1件 = 1カード、枠線色で状態表現）:
  - `☆`スター＋タイトル（例: `☆納骨堂150423・賴髙画也（賴の旧字 髙、注意）・…渋谷区【実相寺】住職紹介`）
  - `おひとりさま` などの属性タグ
  - 区画リンク（`夫婦壇D10-03`）＋法要（`夫婦壇七回忌`）＋ふりがな
  - **`伝言` ピン留め**（赤枠ボックス）= 全員に見せたい申し送りを最上部固定
  - 日付つき履歴の数行プレビュー（`(岩田)` 記録者付き）
  - `⚠ ペット墓` などの注意タグ
- 右下に `＋新規案件`（固定ボタン）。

**示唆:**
- **検索＋タグ＋状態フィルタを最上段に固める** IA は我々のダッシュボード／檀信徒一覧にそのまま効く。`lib/search` 未実装の穴をここで埋める価値が高い。
- **「伝言ピン留め」＝チーム申し送り** は `InteractionNote` に `isPinned` を足せば実現可能。電話応対時に真っ先に目に入るべき情報。
- **自由タグ（確認事項あり／心配／高齢夫婦）** で横断抽出 = 我々もタグ機能を持つべき（後述 §7 改善案）。
- カードのタイトルが「全部入り（場所・氏名・読み・紹介経路・注意書き）」なのは検索性重視の現場知。我々は構造化フィールドで同等の一覧情報密度を出す。

## 3. 檀家カルテ（フレーム2〜6）— 1ページ集約型

世帯（家）の全情報を1スクロールに集約。上から:

1. **概要ヘッダー**: `区分（信者/檀家等）` / `区画・お壇（リンク D02-03）` / `メモ`（自由記述: 「夫婦壇3体契約」「当主の会田様は独身」「最終連絡先は弟さんですが年齢は近い」等の生々しい運用メモ）
2. **家族テーブル**: 列 = `氏名(リンク)` `当主(○)` `役職` `住所[地図]` `電話番号` `日中連絡番号` `メール` `続柄` `年齢` `立ち位置` `埋葬予定場所` `[削除]`
   - **`立ち位置` が色チップ**: `契約者`(緑) `使用者`(青) `連絡先`(灰) `最終連絡先`(紫)。＝1人が複数のロールを持つ関係性の可視化。
   - 住所末尾に `[地図]` インラインリンク。
3. **故人テーブル**: 列 = `俗名(リンク[編集])` `戒名` `続柄` `没年月日` `行年` `納骨年月日` `埋葬場所`。
   - 各故人の下に **`回忌` 行が展開** され、`没後13年` `49`忌 `100`忌 `1`〜`100`回忌 までを **和暦＋西暦＋丸数字バッジ**（49,100,1,3,7,13,17,23,27,33,50,100…）で横並び表示。＝年忌計算結果のビジュアル化。
   - `[遺骨の受入]` などのアクションリンク。
4. **カルテ内タブ**: `伝言 | 家 | 会費 | 契約 | 法要 | 履歴 | 画像`（セクション間ジャンプ）。
5. **法要セクション**: `担当者` `法要名(リンク[編集])` `法要年月日（時間レンジが時刻リンク 10:30〜11:30）` `故人`。
6. **履歴情報（タイムライン）**:
   - **色分けラベルチップ**: `契約`(橙) `法要`(緑) `葬儀` `見学` `家族` `健康` `伝言`(赤) `予定`(青) `お参り` `戒名` ＋`[ラベル編集]`。チップをクリックで**その種別のみ絞り込み**（選択中は黄枠）。
   - テーブル: `日付` `ラベル` `状況（自由記述・長文OK）` `入力者` `[編集][削除]`。
   - 記述が極めて具体的（「堀越さんにLINEで連絡したけど既読にもならない」等）＝**属性DBではなく物語の記録**。
7. **画像・書類保存**: ファイル一覧（PDFサムネ＋`使用申込書[編集]`＋`更新日時`）。
8. **関連する家** / `親族追加` / `家追加` ボタン。
9. **画面下部に固定アクションバー**:
   ```
   履歴登録 | スケジュール | 伝言登録 | 顧客登録 | 法要登録 | 画像保存 | 区画・お壇予約 | 家登録 | 契約登録 | 削除
   ```

**示唆（最重要）:**
- **「カルテを開いた瞬間に家族背景・約束・困りごとが一望できる」面会前プレビューUX** が体現されている。我々の檀信徒詳細はこの1ページ集約思想に寄せるべき。
- **`InteractionNote` のUI実装（最大の欠落）の理想形がこれ**: 色ラベル＋ラベル絞り込み＋記録者＋自由長文＋ピン留め伝言。
- **年忌の丸数字バッジ＋和暦西暦併記** は `lib/nenki`/`lib/wareki` 資産で即再現可能。視認性が高くお寺の人に刺さる。
- **立ち位置（ロール）チップ** = `Person` に役割（契約者/使用者/連絡先/最終連絡先）を持たせる発想。我々の世帯モデルに無い概念で、追加価値が高い。
- **下部固定アクションバー** はスマホでもPCでも有効。我々の詳細画面に「主要操作を常時手の届く位置に」固める。

## 4. 登録モーダル（フレーム6, 66:00）

故人／檀信徒の登録は**モーダルのフルフォーム**。フィールド:
`郵便番号`（+`郵便番号から住所を入力`ボタン）/ `都道府県`(select) / `市区町村` / `丁目番地` / `建物名` / `役職`(select) / `DM郵送可`(checkbox) / `バースデーカード郵送可`(checkbox) / `生年月日`(+`不明`) / `故人`(checkbox) / `戒名`(+`梵字`入力ボタン) / `戒名かな` / `戒名授与者`(select) / `没年月日`(+`不明`) / `行年（満年齢）`(+`不明`) / `行年（数え年）`(+`不明`) / `備考`(textarea) / `登録`・`キャンセル`。

**示唆:**
- **`不明` チェックボックス** が随所にある = 過去帳の現実（没年不明・行年不明）に対応。我々の `datePrecision`（FULL/YEAR_MONTH/YEAR/UNKNOWN）と同じ思想で、UIにも「不明」トグルを出すべき。
- **`郵便番号→住所自動入力`**、**`行年 満年齢/数え年` 両持ち**、**`梵字`入力支援**、**`DM/バースデーカード郵送可` フラグ** は実務的。郵送可フラグは案内状差込（E10）の宛先抽出に直結。
- ただしフィールド数が多くモーダルが縦長。**我々はセクション分け・スマホ縦動線**で入力負荷を下げる。

## 5. 区画マップ（フレーム7, 70:00）

墓地・お壇を**ビジュアルな区画タイル**で表示。各タイル = `区画種別（特別壇/家族壇/個人壇）` ＋ `区画番号（A02-00, B10-00…）` ＋ `使用者姓（坂本/塩川/保坂…）`。空き区画は空タイル。クリックで詳細へ。

**示唆:**
- 我々の**区画地図DnD資産（`kukaku`）が方向性として正しい**ことの裏付け。タイルに「種別・番号・使用者姓」を出し、空き/使用中を一目で。
- 区画→カルテ、カルテ→区画の相互リンク（カルテ概要の `区画・お壇 D02-03` リンク）で**地図・会計・カルテを分断しない統合ビュー**を実現している。

## 6. 年忌一覧（フレーム8）＋ 集計ビュー（フレーム11）

### 年忌一覧（74:00）
チェックボックス付きテーブル。列 = `☑` `#` `姓(リンク)` `氏名` `（ふりがな）` `住所` `故人` `没年月日` `次の回忌（七回忌 2026-12-18 等を自動算出）` `法要予定（リンク＋日付）`。
- **次の回忌を自動計算して列表示**＋行チェックで**一括選択→案内対象抽出**の動線（=「2ボタンで法要案内」の入口）。

### 集計ビュー（86:00）— E08 の理想形
- `期間`（2025年04月〜2026年03月）＋ `月別 / 科目別` トグル ＋ `CSV出力`。
- **科目（行）× 月（列, 4月〜3月）＋ 年計** のクロス集計表。
  - 行: `行事収入`(お布施/塔婆) `会費収入`(護持会費/墓地管理費) `法要布施` `葬儀布施` `契約金` `その他収入`(賽銭) `合計`。
  - **金額セルがリンク**（クリックで明細一覧へ）。`合計` 行・`年計` 列をハイライト。
- 会計年度が**4月始まり**（宗教法人会計の慣行）。

**示唆:**
- **E08 会計帳票はこのクロス集計（科目×月×年計）をそのまま目標形にできる**。`Transaction` を科目分類して集計、4月始まり、セル→明細ドリルダウン、CSV出力。
- **年忌一覧の「チェック→一括抽出→案内」動線** は E10（案内状差込）と E35（郵便振替）に直結。我々の郵便振替も「対象を選んで一括出力」の入口を年忌一覧/会費一覧に置くべき。

## 7. 我々のアプリへの反映（優先度つき改善バックログ）

動画解析から導いた具体的UI/UX改善。既存エピックに対応づけ。

### P0（カルテ体験の核・最大の欠落を埋める）
1. **対応履歴タイムラインUIの実装** — `InteractionNote` に色ラベル分類＋ラベル絞り込み＋記録者表示＋自由長文。檀信徒詳細に時系列表示。（既存 `interaction-queries.ts`/記録者機能を土台に）

3. **年忌の丸数字バッジ＋和暦西暦併記** 表示（`lib/nenki`/`lib/wareki` 流用）。
4. **下部固定アクションバー**（詳細画面）— 履歴登録/法要登録/スケジュール等を常時手元に。スマホ・PC両対応。

### P1（運用効率・抽出導線）
5. **グローバル検索バーの常設**（`lib/search` 実装）— 氏名・かな・TEL・メモ横断のあいまい検索。
6. **自由タグ機能** — 「確認事項あり/心配/高齢夫婦/合祀予定」等を世帯・案件に付与し横断抽出。
7. **伝言ピン留め**（`InteractionNote.isPinned`）— 申し送りを詳細・一覧の最上部に固定。
8. **立ち位置（ロール）チップ** — `Person` に契約者/使用者/連絡先/最終連絡先ロール。
9. **会計クロス集計ビュー（E08）** — 科目×月×年計、4月始まり、セル→明細、CSV出力。

### P2（入力負荷・実務フラグ）
10. **「不明」トグルUI** — 没年月日・行年に不明チェック（`datePrecision` をUIに露出）。
11. **郵便番号→住所自動入力**、**行年 満/数え 両持ち**、**郵送可フラグ（DM/バースデーカード）**。
12. **区画タイルに種別・番号・使用者姓**、空き/使用中の色分け。カルテ⇄区画相互リンク。

### やらないこと（特許・方針）
- **「更新順ポータルの自動並べ替え」を必須挙動として実装しない**（特許第7282407号回避）。タイムラインは日付降順などユーザー選択のソートに留める。
- **15個の横並びタブを丸写ししない** — 次世代・スマホ両対応のため機能をグルーピングし、PCは密度／スマホはスタックの2レイアウトで出す。
- 高齢者特化に寄せすぎない（らくらくスマホ化しない）方針は維持しつつ、本動画の「情報密度の高い1画面集約」をモダンに再構成する。

## 8. デザインの見た目メモ（意匠の参考）

- 配色: ヘッダー／テーブル見出しは**ティール〜濃青のバー**、本文は白地。ラベルチップで彩度の高い色を点で使う。
- テーブル中心の高情報密度。見出し行はグレー／濃色ベタ＋白文字。
- リンクは青、操作系（編集/削除）は `[ ]` 囲みの小リンク。
- アクションボタンは角丸の青系ベタ。
- 我々の方針: この情報密度を活かしつつ、**温かみ（やや暖色のアクセント・余白・角丸・寺院らしい上品さ）** と **モダンSaaS（shadcn ベースの整ったコンポーネント）** を両立。PC＝テーブル密度、スマホ＝カード/スタック。
