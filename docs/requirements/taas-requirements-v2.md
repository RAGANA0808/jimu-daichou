# 寺務台帳 TaaS 要件体系 v2（Temple as a Service としてあるべき要件）

最終更新: 2026-06-07
出典: せいざん「寺務台帳」概要HTML 全12セクション＋スクリーンショット22枚／自社コード棚卸し（`apps/web/src/` 直接Grep・`prisma/schema.prisma` 全734行精読）／`docs/requirements/feature-backlog.md`（E01〜E35）／`docs/legal/patent-analysis.md`
作成: プロダクトアナリスト（網羅性クリティック反映版）

> せいざんを包含しつつ、TaaSとして必要な要件を領域別に体系化。各要件に **優先度**（P0=自寺運用に直結・北極星の核 / P1=業界必須・Phase1 / P2=差別化・Phase1後半〜Phase2）と **由来タグ**（`せいざん由来` / `独自`）を付す。せいざんに無いが必須のもの（権限実効化・監査・通知・モバイル・移行・バックアップ）を明示的に含める。
> **本v2での主なクリティック反映**:
> - C-3「第2連絡先フォーム入力(P0)」は**削除**（入力欄は既に存在＝実装済を確認。`seizan-coverage-matrix.md` §0.1）。残課題はC-2の構造化に一本化。
> - D-1にDocument紐付け列追加を要件化（householdId?のみ＝区画/会計/過去帳に未対応）。
> - refresh_token暗号化(P-6)を**双方向同期から切り離し前倒し**（平文保存の保全リスク）。
> - T-4 巡回は「**順序付きリスト並べ替え**」に限定し**地図経路描画はしない**（P8特許回避）。
> - 承継履歴モデル(C-7)・楽観ロック(M-5)・ShipmentRecipient突合キー(A-2強化)を追補。

凡例（現状）: ✅実装済 / 🟡部分実装 / ❌未実装
