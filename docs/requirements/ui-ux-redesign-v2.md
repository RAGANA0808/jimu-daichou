# C) UI/UX 再設計方針

## C-0. デザイン原則（北極星のUI翻訳）
- **「すぐ見つかる」**: グローバル検索バーをヘッダー常設（実装済）。2タップ以内で結果到達。
- **「気づく」**: ダッシュボードは静的リンク集でなく**期限ベースの行動カード**（cockpit）。
- **「思いやる」**: カルテを開いた瞬間に家メモ＋直近履歴＋次回年忌が見える（面会前プレビュー）。
- **下限としてのアクセシビリティ**: 16px基準・行間1.6・44pxタップ・WCAG AA・色のみに依存しない（文字+アイコン+色）。高齢者特化に寄せすぎない（中心は次世代の若い住職）。

## C-1. せいざんUXパターン → 自社UIへの落とし込み

| せいざんUXパターン | 自社UIへの翻訳 | 意匠・手段の差別化 |
| :-- | :-- | :-- |

## C-2. 詳細ページのタブ化（UI-4）
カルテ詳細を「基本／会費・契約／法要・履歴／区画・納骨／書類」のタブへ。各タブ最上部に**家メモ**を常駐。せいざんの1ページ縦長集約を、情報過密を避けつつタブで再構成（モバイルでも迷子にならない）。

## C-3. ダッシュボード cockpit（更新順ポータル回避の要）
- **主軸は期限ベースの気づきカード**: 今月の法要 / 直近年忌対象 / 未収護持会費・管理料 / 合祀移行間近 / 契約期限到来 / 空き区画 / 最新対応履歴。
- **データ品質KPIカード**（独自・せいざン超え）: 住所欠損率・連絡先欠落率・年忌案内漏れ・未収率。
- **明確にやらないこと**: 「固有情報の更新を契機に契約情報を更新順でポータル自動再配列」する挙動（せいざん特許第7282407号 請求項(3)(4)の核）。更新順は任意ソートの一選択肢に留め、既定にしない。**経年トレンドも「集計値の折れ線/棒グラフ」に限定し、最近更新された世帯/契約を時系列で並べる行リストをダッシュボード主構成にしない**。

## C-4. 特許・意匠・商標の回避線（UIに直結する部分）

| 回避対象 | 根拠特許/商標 | UI/設計上の回避線 |
| :-- | :-- | :-- |
| ブランド配色 | 意匠（見た目） | **ティール/シアンを使わない**。温かい橙 `--brand:#c2410c` を維持。区画ステータス色（空き=グレー/使用=青等）はブランド色ではなく機能色として別系で運用 |
| 更新順ポータル自動並べ替え | せいざん 特許第7282407号 / JP2026042974A | ダッシュボード/タイムラインを**更新順自動再配列にしない**。期限ベース・日付降順固定・任意ソート |
| 見込み客＋契約情報の更新順ポータル | せいざん P1/P2 | 見込み客は属性タグ/ステータスとして既存モデルに保持。**専用ポータル画面を作らない**。通常の検索/フィルタ結果で提供 |
| 施主死亡→次施主自動設定 | TERA Tech 特許第7705103号 | 承継は**全自動にしない**。死亡登録は「承継候補の提示・通知」まで、確定は**担当者の明示承認（手動）**を必須。承継はC-7履歴モデルに証跡を残す |
| 命日基準スケジュール＋オンライン参拝/決済 | Classix JP7291923B1 | 中陰/年忌計算は**純粋ロジックに独立**（R1公知で防御）。**オンライン参拝・決済・仮想寺院・納骨箱搬送と連結しない**。通知は寺側内部に限定 |
| **【追補】地図上の墓参/巡回ルート登録** | **YMFG JP2022036640A（P8）** | **巡回(T-4/E30)は地図に経路ラインを描画・登録しない**。訪問先の**順序付きリスト並べ替え**とシフト表帳票に限定。区画マップは色分け＋台帳逆引きのみ |
| 遠隔墓参・墓ライブ配信・IC認証故人表示 | P7〜P10 | 区画マップは**色分け＋台帳逆引き**に限定。**撮像/遠隔配信/IC認証はスコープ外** |
| 名称「檀信徒カルテ」「寺務台帳」 | 商標 P3/P4 | **対外サービス名/機能名/マーケ表現としてそのまま採用しない**ことを検討。内部コード通称（CLAUDE.md）と対外ブランドを分離。確定前に弁理士FTO（L-1/L-2） |

# D) 実装ロードマップ（ウェーブ分割）

> 既存資産は継続・拡張（作り直さない）。全変更は tenantId+RLS+withTenant・過去帳論理削除・JST・lib集約を順守。スキーマ変更は必ずマイグレーション。**独立性の高いもの**と**相互依存（権限・基盤）**を区別する。

## Wave UI-4 — 構造刷新（並行可・スキーマ変更なし）
- **含む**: C-1/C-2(検索)サジェストランキング(S-2)、カルテ詳細のタブ化(C-2)、一覧テーブル⇄カードの全画面徹底(M-1)、ダッシュボード cockpit の期限ベースカード＋未収/管理料サマリ連携(Z-1)、タイル情報密度（種別+番号+使用者姓、フロントのみ）、年忌バッジに「あと◯年」明示(N-10)、家メモのカルテ最上部昇格(C-4)、戒名順ソート(K-3)・一覧CSV導線(K-4)。
- **スキーマ変更**: 不要。
- **再利用資産**: 共通UIキット(`apps/web/src/components/ui/`)、`features/dashboard/queries.ts`、`features/bochi`/`gojikai`の集計クエリ、`lib/search`、`features/kakochou/NenkiBadges.tsx`、`features/kakochou/KakochouListControls.tsx`。
- **受け入れ条件**: 全画面でPC高密度/スマホカード出し分け。ダッシュボードに期限ベース＋未収カード。更新順自動再配列は無し。typecheck/test/`tenant-check` グリーン。
- **独立性**: 高（既存データで完結）。

## Wave P0-FOUNDATION — 保全・即効（一部独立・最優先）
- **含む**: P-4 バックアップ/リストア/DR/可搬性(E26, **P0・せいざん必須・過去帳100年保全の根幹**)、Z-2 データ品質KPIカード(独自・せいざン超え)。
- **削除（クリティック反映）**: 旧「第2連絡先フォーム入力(C-3)」は**実装済を裏取りしたため除外**（`HouseholdForm.tsx`に入力欄が存在）。連絡先の残課題は構造化のみで Wave CONTACT-SEARCH のC-2へ移管。
- **スキーマ変更**: E26は不要（運用・ADR・runbook中心）。Z-2はクエリ追加のみで不要。
- **再利用資産**: Supabase自動BU、`lib/export`(可搬性)、`features/dashboard`、`features/bochi`/`gojikai`の集計。
- **受け入れ条件**: 日次BU取得＋監視通知、PITR有効化、リストア演習記録、テナント全書き出し。品質KPIがダッシュボードに表示。
- **独立性**: 高（E26はインフラ独立、Z-2はUI独立）。

## Wave PERMISSION — 権限・監査・保全基盤（相互依存・他に先行すべき土台）
- **含む**: P-1 RBAC実効化(E21)、P-2 ユーザー管理UI(E21)、P-3 横断監査ログ基盤(E25)、**P-6 refresh_token の at-rest 暗号化（双方向同期から切り離し前倒し）**。
- **スキーマ変更**: 要（User拡張、`AuditLog`モデル新設、`Tenant.googleRefreshToken`の暗号化方式変更、マイグレーション）。
- **再利用資産**: `lib/auth/session.ts`、`lib/db/`(withTenantラッパーに監査フック)、E06/E08の既存断片記録を統合、`app/api/google/auth/callback/route.ts`。
- **受け入れ条件**: READ_ONLYで作成/編集/削除が不可。ユーザー招待/無効化UI。閲覧/変更/削除/書出をAuditLogに一元記録（**個人情報平文を載せない**）。refresh_tokenが暗号化保存。
- **独立性**: 低（E21とE25はセット。後続の不可逆操作=合祀/改葬/会計取消/承継承認が本基盤に乗る）。**他ウェーブより先に土台化推奨**。

## Wave CONTACT-SEARCH — 連絡先構造化と全文検索（相互依存）
- **含む**: C-2 ContactPointモデル(多階層連絡先・E24/E22)、S-3 全文検索(E28・pg_trgm)、S-2 サジェスト強化、C-6 形式正規化バリデーション、C-7 承継履歴モデル(HouseholdSuccession)。
- **スキーマ変更**: 要（`ContactPoint`新設＋Household/Personに1対多、pg_trgm GINインデックス、`HouseholdSuccession`新設、マイグレーション）。既存 `secondaryContact`(1行)からContactPointへの移行スクリプトを併設。
- **再利用資産**: `lib/search/normalize.ts`/`queries.ts`、`features/danshintoto`、CSV import/export(連絡先列の往復)。
- **受け入れ条件**: 孫世代まで人数無制限の連絡先を構造化保持・編集。履歴本文/過去帳備考/家族名を横断全文検索＋スニペット。承継が履歴として残る。個人情報非ログ。
- **独立性**: 中（ContactPointは独立、全文検索はContactPoint/履歴に依存。承継承認はPERMISSION前提）。

## Wave GRAVE-CORE — 区画コア（芋づる欠落の根を断つ・最重要依存）
- **含む**: G-3 Burialモデル・納骨使用者自動表示（区画→故人/カルテ→故人 双方向）(E09)、G-2 苗字/墓標名逆引き（刻名カラム＋検索統合、§1/§9/S-4共通）、G-4 GraveContract契約構造化(預かり年数/満了/更新)、G-7 空き区画検索、G-6 ステータス拡張＋地図色分け。
- **スキーマ変更**: 要（`Burial`中間モデル＝tenantId/personId×gravePlotId/納骨日/論理削除、GravePlotに墓標名/刻名、`GraveContract`新設、GravePlotStatus enum拡張、マイグレーション）。
- **再利用資産**: `features/kukaku/*`(地図DnD・区画CRUD)、`lib/search/queries.ts`(`search()`にGravePlotブロック追加=既存2ブロックと同型パターン)、`lib/import/entities/gravePlot.ts`。
- **受け入れ条件**: 区画詳細に納骨者一覧／カルテにそのお墓の故人一覧。横断検索で苗字/墓標名→区画特定。契約満了が機械可読。空き区画検索＋割当。地図がライフサイクル色分け。
- **独立性**: 中（Burial→苗字逆引き→契約→合祀の順で芋づる。**合祀(E10)の前提**）。

## Wave MEMORIAL-EXPIRY — 合祀・改葬・年忌帳票・法要精度（GRAVE-CORE依存）
- **含む**: G-5 合祀期限自動計算→対象抽出(E10)、G-8 改葬/墓じまい/承継ウィザード(**手動承認＝TERA特許回避／C-7履歴に記録**)、N-2 年忌表帳票(戒名順CSV/印刷)、N-6 法要終了時刻、A-2 重複案内防止(突合キー構造化＝ShipmentRecipientに targetPersonId/anniversaryNo)。
- **スキーマ変更**: 一部要（承継/改葬モデル、MemorialServiceにendTime、ShipmentRecipientに突合キー列、いずれもマイグレーション）。N-2は不要。
- **再利用資産**: GraveContract/Burial(GRAVE-CORE)、`findAnniversariesForYear`、`features/shipment`(発送)、`lib/pdf`、`lib/google/calendar.ts`(期限通知)、C-7承継履歴。
- **受け入れ条件**: 安置期間から合祀移行候補を自動リストアップ（案内漏れ防止）。承継は死亡登録→候補提示→**手動承認**。年忌表を戒名順でCSV/印刷。重複案内警告が既送突合で機能。
- **独立性**: 低（合祀/改葬/承継はGRAVE-COREのBurial/GraveContract＋PERMISSION/C-7に依存。帳票系は独立）。

## Wave PAPERLESS-MOBILE — 書類保管・音声・モバイル（一部独立）
- **含む**: D-1 書類/写真クラウド保管(E17・Supabase Storage)、D-2 スマホ撮影保存、D-4 書類論理削除、M-2 音声入力(E19)、N-7 シンプル登録ウィザード、N-8 寺全体行事、N-9 横断タイムライン統合、M-5 楽観ロック。
- **スキーマ変更**: **要（クリティック反映）**。E17はDocumentモデル既存だが **紐付け列が householdId? のみ → gravePlotId/transactionId/deathLedgerEntryId/uploadedById/deletedAt 追加が必須**（ドラフトの"スキーマ変更不要"は誤り）。寺行事は householdId任意化 or TempleEvent新設。
- **再利用資産**: `prisma/schema.prisma`(Document定義済)、`features/danshintoto`/`kukaku`/`kaikei`(添付先)、`features/houyou`(行事)。
- **受け入れ条件**: カルテ/区画/会計/過去帳に書類添付・一覧・プレビュー（テナント分離・最小権限・論理削除）。長文フィールドに音声入力。寺行事を世帯非依存で登録。
- **独立性**: 中（E17は紐付け先モデル(GRAVE-CORE等)とPERMISSIONの監査連携があると望ましい）。

## Wave OUTREACH-NOTIFY — 通知・巡回・配信・設定（Phase2寄り）
- **含む**: T-2 双方向同期(E15)、T-3 自動リマインド(**寺側内部限定＝Classix回避**)、T-4 月参り/棚経巡回(E30・assignedUserId活用・**順序リストのみ／地図経路描画なし＝P8回避**)、A-5 マルチチャネル配信(E14)、O-2/O-3 宗派プリセット/初期設定(E29)。
