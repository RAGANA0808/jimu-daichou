# 寺務台帳 機能要件定義書（SaaS開発用詳細版）

## 1. はじめに
本ドキュメントは、せいざん株式会社が提供するお寺向けSaaSシステム「寺務台帳」を模した、あるいは機能拡張したシステムの機能要件・アーキテクチャを定義するものです。提供されたサービスサイトの情報およびYouTube動画の分析結果に基づき、システムの目的、解決する課題、主要機能、画面構成、操作フロー、データベースの主要項目、および開発上の非機能要件を整理します。

## 2. システムの目的と解決する課題
「寺務台帳」は、単なる檀家管理ソフトではなく、「檀信徒カルテ」として、寺院と檀信徒の関係性を深化させ、持続可能な寺院運営を支援することを目的としています。具体的には、以下の課題を解決します。

*   **情報の属人化と継承の困難:** 住職や特定の寺族に依存していた檀家情報をデータ化し、寺院全体で共有・継承可能な資産とします。
*   **コミュニケーションの質的向上とトラブル防止:** 檀信徒との過去のやり取りや個人的な情報をカルテとして蓄積。言った言わないを防ぎ、一人ひとりに寄り添った対応を可能にします。
*   **事務作業の負担軽減:** 法要案内、年忌表作成、宛名印刷などの定型業務を自動化します。
*   **アナログ管理の限界:** 紙媒体や旧来のシステムによる検索性の低さ、データ消失リスクを解消し、クラウドベースでの一元管理を実現します。
*   **檀家の高齢化と社会変化への対応:** 墓じまいや合祀へのスムーズな移行など、最新の情報を迅速に把握・活用できる仕組みを提供します。

## 3. 主要機能
SaaSとして提供するための主要機能は以下の通りです。

| カテゴリ | 機能概要 | 詳細 |
| :--- | :--- | :--- |
| **SaaS基盤** | マルチテナント管理 | 複数の寺院（テナント）が利用できるよう、データを論理的に完全に分離する。 |
| | 権限（ロール）管理 | 住職（全権限）、副住職（全権限）、一般スタッフ/事務員（会計閲覧不可・削除不可など）のアクセス制御。 |
| **檀信徒管理** | 檀信徒カルテ | 施主、親族、過去帳、新規のご縁を統合管理。家族構成、住所、第2連絡先（緊急連絡先）、お墓の場所を一画面で集約表示。 |
| | 高速検索機能 | 名前（ひらがな可）、電話番号の一部などで即座に該当情報を抽出（インクリメンタルサーチ）。 |
| | 対応履歴（伝言メモ） | 電話内容、訪問記録、法要の相談など、誰がいつどんな対応をしたかを時系列で記録・共有。タグ（ラベル）付け機能。 |
| **法要・行事** | スケジュール管理 | 法要の予約、実施状況、参加人数、塔婆の有無の記録。 |
| | カレンダー自動連携 | システム上で登録した法要スケジュールを、寺院のGoogleカレンダーへ自動追加（単方向同期を想定）。 |
| **帳票・出力** | 書類・帳票自動生成 | 過去帳からの自動計算による「年忌該当者」のリストアップ。法要案内状の宛名ラベル印刷（A4シール紙）、塔婆読上帳（PDFリスト）の出力。 |
| **お墓管理** | 区画マップ管理 | 墓地のレイアウトに基づいたグリッド表示で、空き区画（グレー）や使用状況（青色）を視覚的に把握。クリックで施主情報へ遷移。 |
| **会計管理** | 会計・管理費管理 | 護持会費やお布施等の納入状況（既納・未納・前払い）の管理。ダッシュボードでの売上・未収金のグラフ表示。 |
| **データ移行** | CSVインポート | 既存のエクセル管理データ（施主名簿・過去帳）をシステムへ一括流し込みする機能。導入ハードルを下げるために必須。 |

## 4. UI/UX設計原則（高齢者・現場対応）
ITリテラシーが高くない高齢の住職や寺族でも直感的に操作できるよう、以下の原則を厳守してUIを構築します。

1.  **視認性の確保:** 基本フォントサイズを大きく（16px以上ベース）、行間を広く確保する。
2.  **明確なアクションボタン:** 登録（青）、削除（赤）、キャンセル（グレー）など、ボタンの色と大きさを統一し、誤操作を防ぐ。
3.  **和暦・西暦の相互変換サポート:** 没年月日などの入力では「昭和・平成・令和」の和暦入力をサポートし、システム内で西暦に自動変換・計算させる。
4.  **レスポンシブデザイン:** 本堂や墓地など、PCのない場所でもスマートフォン・タブレットからカルテの閲覧・音声メモ入力ができるように最適化する。

## 5. 画面構成
システムは以下の主要な画面で構成されます。

*   **メインダッシュボード:** 本日の予定（カレンダー連携）、直近の年忌予定者、未収金アラート、対応履歴の最新タイムラインを表示。
*   **グローバル検索バー:** 全画面上部に常設。ひらがな等で施主・故人を即座に検索可能。
*   **世帯詳細画面（檀信徒カルテ）:**
    *   **上部:** 世帯主情報、第2連絡先（親族など）、現在のステータス。
    *   **タブ切替部:** 「故人一覧・年忌（自動計算）」「対応履歴（タイムライン）」「契約・区画」「会計情報」「ファイル保存（スキャン画像等）」をタブで整理。
*   **区画管理画面:** エクセルライクなグリッドマップ表示。
*   **統計・分析画面:** 新規登録数、法要実施率、年代別分布などをグラフ表示。

## 6. 操作フロー（例：檀信徒からの電話での法要相談）
1.  **クイック検索:** 檀信徒からの電話連絡時、上部の検索バーに名前（ひらがな）や電話番号を入力し、1秒でカルテを開く。
2.  **状況把握:** カルテ内の「対応履歴」タブで前回の会話内容を確認。「故人一覧」タブで直近の年忌該当者を確認しながら応対する。
3.  **スケジュール登録:** そのまま「法要追加」ボタンを押し、日時や塔婆の希望を入力。
4.  **自動共有:** 登録と同時に、Googleカレンダーに予定が同期され、他のスタッフのダッシュボードにも通知される。
5.  **履歴記録:** 電話後、対応履歴タブに「〇〇家より13回忌の予約受付。足が悪い方がいるため椅子を用意する」とメモを残す。

## 7. データベースの主要項目 (RDBスキーマ案)

システム構築のコアとなる主要テーブルとリレーションです。

| テーブル名 | 概要 | 主要カラム |
| :--- | :--- | :--- |
| **Tenant** | 寺院（契約者）情報 | `id`, `name`, `plan`, `created_at` |
| **User** | アカウント・権限 | `id`, `tenant_id`, `role`(admin/staff), `name`, `email`, `password_hash` |
| **Household** | 施主・世帯情報 | `id`, `tenant_id`, `name`, `name_kana`, `address`, `phone`, `emergency_contact_name`, `emergency_contact_phone`, `status`(檀家/見込み等) |
| **Deceased** | 過去帳・故人情報 | `id`, `household_id`, `kaimyo`(戒名), `zokumyo`(俗名), `death_date`(没年月日), `age_at_death`(行年), `relationship` |
| **Interaction**| 対応履歴（カルテ）| `id`, `household_id`, `user_id`(対応者), `type`(電話/訪問等), `content`(内容), `tags`, `created_at` |
| **Event** | 法要・行事スケジュール| `id`, `tenant_id`, `household_id`, `title`, `event_type`, `start_time`, `end_time`, `google_event_id` |
| **Plot** | お墓・区画管理 | `id`, `tenant_id`, `household_id`, `plot_number`, `type`, `status`(空き/使用中) |
| **Transaction**| 会計・入出金 | `id`, `household_id`, `category`(護持会費/布施等), `amount`, `payment_date`, `status` |

## 8. 非機能要件・技術スタック要件（推奨）
AIコーディング（Claude Code等）で迅速かつ堅牢に開発するための推奨スタックです。

*   **フロントエンド:** Next.js (App Router), React, TypeScript
*   **UIフレームワーク:** Tailwind CSS, shadcn/ui (シンプルでアクセシビリティの高いUIコンポーネント)
*   **バックエンド / データベース:** Supabase (PostgreSQL), Prisma ORM
*   **認証:** Supabase Auth
*   **帳票出力:** react-to-print などのクライアントサイドPDF生成ライブラリ
*   **外部連携:** Google Calendar API
