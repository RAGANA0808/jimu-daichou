- **年忌計算は「没年」基準で year のみあれば動く**設計にした。月日不明の故人でも年忌年の抽出はできる（2/29 補正は月日が揃っているときだけ発火）

- **テストの致命的ミスを 1 件検出**: 「該当者ゼロ年」として `2024` を指定したが、d の七回忌が 2024 だった。→ `2023` に修正。こういう日付算数ミスこそテストで早期発見できてよかった

## 午後: RLS 多層防御の完成

※ ログ前半の「次回やること」では RLS が未着手扱いだったが、実際は既に `withTenant()` ラッパー・マイグレーション・統合テストが揃っていた (ログ更新漏れ)。午後のセッションで整合性を取り、**実運用上 RLS が効く状態** までを完成させた。

- **統合テストが重要な抜けを検出**
  - `pnpm test:integration` で 5 テスト中 3 つ失敗 → 原因は Supabase の `postgres` ロールが `BYPASSRLS` で接続していたため、ポリシーは適用済みでも素通りしていた
  - 「RLS 適用済み」と「実効的に効いている」は別物、と実感できた回
- **アプリ専用 DB ロール `jimu_app` を導入**
  - `scripts/supabase/create-app-role.sql` 追加（Supabase Console で 1 回だけ実行する運用 SQL）
  - 属性: `LOGIN / NOBYPASSRLS / NOSUPERUSER`、`public` スキーマ CRUD + 将来テーブルへの `DEFAULT PRIVILEGES`
  - `DATABASE_URL` を `jimu_app` 経由に、`DIRECT_URL` は `postgres` のまま（マイグレーション実行権限が必要）
- **接続経路を Session Pooler (5432) に変更**
  - Transaction Pooler (6543) では custom role の認証が失敗する事象。Phase 1 では接続数が少ないので 5432 で十分
  - Vercel デプロイ時に接続数がネックになったら 6543 復帰を再検討
- **RLS ポリシーのハーデニング** (`20260423045000_harden_rls_null_handling`)
  - `current_setting('app.current_tenant_id', true)` は同セッション内で一度 `SET LOCAL` すると、以降の別 Tx で空文字列 `""` を返す PostgreSQL の仕様 → `""::uuid` が `22P02` エラーになる
  - 全テーブルのポリシーを `NULLIF(..., '')::uuid` ラップに置換し、NULL と `""` を同じく扱うよう修正 → 未設定時は決定論的に 0 行
- **統合テストを jimu_app 前提に書き直し**
  - `adminPrisma` (DIRECT_URL = postgres 権限) を別クライアントとして追加し、setup / teardown のみ RLS を bypass
  - RLS 境界検証本体は通常 `prisma` (jimu_app) を使用 → **本番 Server Actions と完全に同じ条件で** 多層防御を検証
