## 今日やったこと

### Supabase の "Table publicly accessible" 警告対応

- **発端**: Supabase から「RLS 未有効のテーブルが 1 件あり」のメール警告 (jimu-harness-dev プロジェクト宛)
- **原因特定**: 診断スクリプト `scripts/check-rls.ts` を作って public スキーマの全テーブルの RLS 状態をダンプ → `_prisma_migrations` (Prisma 自動生成のマイグレーション履歴管理テーブル) のみ RLS 未有効と判明。アプリ側の 11 テーブルは全て有効
- **対応**: `prisma/migrations/20260430000000_enable_rls_prisma_migrations/migration.sql` を追加、`pnpm db:migrate` で適用
  - `ALTER TABLE IF EXISTS "_prisma_migrations" ENABLE ROW LEVEL SECURITY;` のみ
  - ポリシーは作らない = anon/authenticated は全拒否、テーブルオーナー (postgres) はバイパス
  - 既存の `20260423022852_enable_rls` と同じ慣習 (FORCE は付けない)
- **検証**: 再度 `check-rls.ts` を実行 → 12 テーブル全て `[OK]`、未有効 0 件

## 決めたこと・気づき

- **`_prisma_migrations` を見落としがちな理由**: Prisma 管理のシステムテーブルなので Prisma スキーマには現れない → 自前の RLS マイグレーション台帳からも漏れる。今回のように「全テーブルを直接 SQL で確認するスクリプト」を 1 本持っておくのが安全
- **実害の評価**: このテーブルに入っているのはマイグレーションファイル名・チェックサム・適用日時のみ。檀信徒データ漏洩には直結しないが、改ざんで Prisma の整合性が壊れる経路になるので Phase 2 (SaaS 化) 前には必修事項
- **`scripts/check-rls.ts` は残す**: 今後新しいテーブル追加時の検査に流用できる。`pnpm tenant-check` の RLS 版として位置づけ
