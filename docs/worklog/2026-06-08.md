### PII 持出 API ルートの権限ガード+監査の統一（アドバーサリアルレビュー由来）

`nenki/csv`（既に `requireCapability('export')` + EXPORT 監査）に対し、同種の PII 持出ルートでガード・監査が欠けていた非対称を解消。**方針はユーザー判断で「全ルート export で統一」**（Phase1 は住職=HEAD_PRIEST バイパスで実害なし、Phase2 の STAFF/READ_ONLY ロールで権限が効く・全持出が監査に残る）。

- **対象8本を `const user = await requireCapability('export')` + 生成成功後に `withTenant(tenantId, tx => recordAudit(tx, tenantId, {action:'EXPORT', entityType:'Export', summary}))` に統一**:
  - `api/shipment/csv`（宛名CSV・最優先med。従来 export ガード/監査ゼロ）/ `api/shipment/pdf`（ラベル/封筒/案内状の3分岐は closure `auditExport()` で集約）/ `api/gojikai/dunning`（督促状）/ `api/bochi/demand`（催告状）/ `api/furikae/pdf`（一括/単票・try内で監査）/ `api/chuin/pdf`（中陰表）/ `api/toba/pdf`（塔婆読上帳）。
  - **参考ルートだった `api/nenki/pdf`（案内状PDF）もユーザー判断で統一**（shipment/pdf notice と機能等価のため非対称を残さない）。
- **監査 summary は PII 非載厳守**: 種別・年度/年度+回・件数・UUID(entryId/memorialServiceId) のみ。氏名/住所/戒名は載せない。
- 既存の `requireCurrentTenantId()` を `requireCapability('export')`（戻り値 `User` の `tenantId`/`id` を再利用）へ置換。テナント分離は従来どおり各クエリ + withTenant で担保され、ガードは厳密に強化のみ。
- **静的ゲート全GREEN**: `pnpm typecheck` ✅ / `pnpm tenant-check` ✅ 違反ゼロ / `pnpm test` ✅ 438件。
