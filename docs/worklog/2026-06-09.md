### ❗ 検証手順の重要な知見(次セッションで時間を溶かさないため)
- **chrome-devtools(CDP)のブラウザ・ナビゲーションは Edge runtime の middleware で `getUser()` fetch が中断("The user aborted a request"→Retrying)され、有効 cookie でも `/login` に 307 リダイレクトされる**。一方 **node/curl からの単発リクエストは 200** で正常レンダリング(middleware も page も Node 経路では正常)。これは環境固有(CDP×Edge middleware)であり**アプリ/コードのバグではない**。
- 回避した検証手順(auth を弱めず): ① `scripts/dev-login.ts` を**自分で実行可能**(.env 読取は Read ツールでは権限ガードされるが、Node スクリプト内の readFileSync は通る)→ cookie 取得。② 認証済み HTML を **node fetch で取得**(`fetch('http://localhost:3104/bunseki',{headers:{cookie}})` が 200)→ 構造アサート。③ 取得 HTML に `<base href="http://localhost:3104/">` を注入し `<script>`/`<template>` 除去・`<div hidden id="S:">` の hidden を解除して**静的SSRを file:// でブラウザ描画→screenshot**。
- やってはいけない(auto classifier がブロック): セッショントークンを web 配信 public ファイルに書く / `JDC_VERIFY_BYPASS` 等で middleware の auth を弱めて dev 起動する。いずれも妥当なガードレールなので回避しない。
- `pnpm tenant-check` は npm script が `bash scripts/tenant-check.sh` を直呼びするため **PowerShell では bash 不在で exit1**。**Bash ツールで `bash scripts/tenant-check.sh` を直接実行**すれば通る(検査ロジックは正常)。
