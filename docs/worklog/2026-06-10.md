リポジトリの地盤固め（ユーザー選択「リポジトリの地盤固め」）。マージ済みリモートブランチの掃除と、GitHub Actions による CI（品質ゲート）整備。

## 今日やったこと

### マージ済みリモートブランチの掃除
- このセッションで作った feature/docs ブランチ群（PR #3〜#22）は GitHub 側ではマージ時に自動削除済みだった。残っていたのは **ローカルの stale な追跡参照 20 本**のみ（`git push --delete` は「remote ref does not exist」で全件 no-op）。
- `git remote prune origin` で追跡参照を掃除。リモートは `origin/main` のみ・tree クリーン。

### GitHub Actions CI（品質ゲート）— PR #23
- `.github/workflows/ci.yml` を新規追加。`main` への push / `main` 向け PR で以下を**単一ジョブ・直列**実行:
  `pnpm db:generate` → `typecheck` → `test`(vitest unit) → `lint` → `tenant-check` → `next build`。
- **Prisma datasource が要求する `DATABASE_URL`/`DIRECT_URL` はダミー注入**（実 DB に接続しない）。
- **build もダミー env 一式で通る**ことを CI 実行で実証（全ルート force-dynamic ＋ Supabase/Google env の getter 遅延評価のため、prerender 中にシークレットを使う実処理が走らない）。dotenv 依存の package `build` を避け `apps/web` で `next build` を直接実行。
- 既存 `supabase-keepalive.yml` と共存。`concurrency` で同一参照の古い実行を自動キャンセル。
- **PR #23 の CI 実行で全ステップ green（2m45s）を確認 → マージ**。main = `cee2869`。

## 決めたこと・気づき
- build を CI に含めるかは「実 env / シークレットが要るか」が論点だったが、**force-dynamic + env 遅延評価のおかげでダミー env で検証可能**と判明。シークレットを CI に置かずにビルド可能性まで保証できる。
- 実 env での本番ビルド検証は引き続き **Vercel の Preview デプロイ**が担う（CI と二重化しない）。
- ⚠️ **Node.js 20 actions の deprecation 警告**（2026-06-16 以降 GitHub が Node 24 を既定化）。`actions/checkout@v4` 等は現行最新メジャーで CI は green・実害なし。将来 actions のメジャー更新時に上げる。
- HANDOFF §2D の「`pnpm lint` は実行不能」「`pnpm build` 完走しない」は PR #20/#18 で既に解消済みだったので記述を是正。品質ゲートの記述も lint を含め CI 化を反映。
